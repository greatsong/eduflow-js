**핵심 원칙:** AI가 만든 코드나 결과물이 아니라, **학생이 설계한 규칙 파일·명세서·수용 기준**을 평가한다.

## 3. 이론적 토대 — 원전 요약

이 교육과정은 아래 원전들이 만나는 지점에 서 있다. 상세 분석은 `07_원전분석_교육과정과의연결.md` 참조.

### 3-1. Hashimoto — "하네스 엔지니어링" (2026.02.05)

Mitchell Hashimoto(Ghostty 개발자)의 AI 도구 채택 6단계 여정.

**교육과정에 직접 영향을 준 핵심:**
- Step 5 "하네스를 설계하라": **"에이전트가 나쁜 짓을 할 때마다, 다시는 못 하게 막는다"** → 규칙 파일 개념의 직접적 출처
- Step 2 "내 일을 재현하라": 일을 두 번 한다(먼저 수동으로, 그 다음 에이전트로) → 교육과정의 [1] 직접 해보기 → [2] AI에게 시키기 → [3] 비교하기
- AGENTS.md의 "각 줄은 에이전트가 실제로 저지른 나쁜 행동에 기반해 추가된 것" → 규칙 파일의 "이 규칙이 생긴 사건" 기록

**교육과정은 Hashimoto가 6개월에 걸쳐 혼자 체득한 것을, 30시간에 구조화된 경험으로 압축한 것이다.**

### 3-2. Fowler — "엄격함의 재배치" (2026.01.07)

Chad Fowler가 소프트웨어 역사(동적 언어, XP, 지속적 배포)에서 반복되는 패턴을 식별한 글.

**교육과정의 숨은 뼈대가 되는 핵심 문장들:**

> **"생성이 더 쉬워진다면, 판단은 더 엄격해져야 한다."**

> **"안쪽은 확률적이고, 가장자리는 결정적이어야 한다."** (probabilistic inside, deterministic at the edges)

교육과정 7개 목표가 이 문장의 풀이다:
- 목표 1~3 (인식/판단) = "판단은 더 엄격해져야 한다"
- 목표 4~5 (구조화) = "의도를 정밀하게 명세하는 일"
- 목표 6~7 (반복 개선) = "규율을 재배치하는" 실천

### 3-3. OpenAI — "에이전트-퍼스트 하네스 엔지니어링" (2026.02.11)

3명의 엔지니어가 Codex로 사람 손코드 0줄인 제품을 5개월 만에 출시한 사례.

**교육과정에 가져온 개념:**
- **"에이전트 관점에서, 컨텍스트 안에서 접근할 수 없는 것은 존재하지 않는다"** → 학생 언어: "AI에게 안 알려준 건 AI가 모르는 거다"
- **"AGENTS.md를 백과사전이 아니라 목차로 취급하라"** → 규칙 파일이 30개 이상으로 자랄 때 범주별 정리가 필요한 근거
- **가비지 컬렉션**: 나쁜 패턴이 퍼지기 전에 주기적으로 정리 → 매 트랙 종료 시 규칙 정리 활동

### 3-4. Böckeler — 하네스의 3요소 분류 (2026.02.17)

Thoughtworks의 Böckeler가 OpenAI 글을 분석하며 하네스를 3범주로 분류.

| 하네스 요소 | 교육과정 대응 |
|------------|-------------|
| 컨텍스트 엔지니어링 | 문제 구조화 능력 (목표 4) |
| 아키텍처 제약 | 수용 기준을 미리 세우는 능력 (목표 5) |
| 가비지 컬렉션 | 반복적 점검과 개선 루틴 (목표 7) |

**중요한 지적:** Böckeler는 OpenAI 글에서 **기능/동작 검증이 빠져 있다**고 비판했다. 교육과정은 "수용 기준"으로 이것을 명시적으로 포함하고 있으며, 이것이 교육과정의 고유한 강점이다.

### 3-5. 분석글: "스캐폴딩과 하네스"

부벡(Bubeck)의 스캐폴딩과 하네스 엔지니어링의 차이를 분석한 글.

- **스캐폴딩** = 인지적 깊이를 향한 장치 (탐색을 조직)
- **하네스** = 조직적 지속성을 향한 장치 (실패의 재발을 차단)

**교육과정에 주는 시사:** 학생이 만드는 **"규칙 파일"은 하네스**이고, **"명세서"(좌표, 크기, 색상 등)는 스캐폴딩**에 가깝다. 학생은 둘을 동시에 배우지만 교육과정에서 이 구분은 명시되어 있지 않다. 선생님이 이 차이를 인식하면 지도에 도움이 된다.

## 6. 가드레일 체크포인트

> 아래 문장들은 이 교육과정의 본질에서 벗어나지 않기 위한 판단 기준이다.
> 문서를 작성하거나 활동을 설계할 때, 이 문장들과 충돌하는지 확인하라.

### 교육의 본질

1. **이 교육의 산출물은 챗봇도, 시뮬레이션도, IoT 장치도 아니다. "규칙 파일"이다.** 도메인(챗봇, VPython, 데이터, Pico)은 규칙 파일을 자라게 하는 토양일 뿐이다.

2. **규칙은 사전에 계획해서 쓰는 것이 아니라, 실패 경험에서만 자라난다.** 선생님이 "좋은 규칙 목록"을 미리 제공하는 순간, 교육의 핵심이 죽는다.

3. **AI가 실수하는 것은 수업의 실패가 아니라 학습의 엔진이다.** "AI가 이상한 결과를 줬다" = "잘됐다, 규칙 하나 더 생겼네!"

4. **평가 대상은 AI가 만든 것이 아니라, 학생이 설계한 구조다.** 명세서, 수용 기준, 규칙 파일, 과정 기록, 성찰 에세이가 평가 대상이다. AI가 만든 코드의 품질은 평가하지 않는다.

### 설계 원칙

5. **"생성이 더 쉬워진다면, 판단은 더 엄격해져야 한다." (Fowler)** 교육과정의 모든 활동은 이 문장의 풀이여야 한다.

6. **학생은 먼저 직접 해보고, 그 다음에 AI에게 시킨다.** AI 없이의 기준선이 없으면 AI 결과를 판단할 수 없다. "직접 해보기"를 생략하면 수용 기준을 세울 근거가 사라진다.

7. **수용 기준은 결과를 보기 전에 세워야 한다.** 사후에 세우면 기준이 결과에 끌려간다. 이것은 교육과정 전체에서 반복 강조되는 원칙이다.

8. **규칙을 만드는 방법은 같지만, 검증 방법은 도메인마다 다르다.** 이것이 30시간 후 학생이 도달해야 할 메타 인사이트다. 트랙별 활동이 이 인사이트를 향해 수렴해야 한다.

### 교실 운영

9. **종이를 써라.** 화면에서 바로 수정하면 "생각 없이 복사"가 된다. 종이에 먼저 쓰고, 정리한 다음 코드/문서에 옮기는 과정이 의식적 설계 훈련이다.

10. **남이 찾아준 문제가 가장 가치 있는 규칙을 만든다.** 자기 봇은 자기가 아는 방식으로만 테스트하기 때문이다. 교차 테스트와 갤러리 워크를 충분히 활용하라.

11. **주제 선정에 5분 이상 쓰지 않게 하라.** "완벽한 주제"를 찾으려다 시간을 낭비한다. 주제는 규칙을 자라게 하는 토양일 뿐이다.

12. **도입부를 절대 생략하지 마라.** "실패 → 규칙 → 재시도" 루프를 체험하지 않으면, 이후 트랙에서 규칙 파일을 만드는 행위의 의미를 이해하지 못한다.

### 기술적 판단

13. **AI에게 안 알려준 건 AI가 모르는 거다.** SYSTEM_PROMPT를 "대충" 쓰면 봇이 망가지는 경험이 이 원칙을 체득시킨다.

14. **"뭘 시킬지 정확히 정하는 게 실력이다."** 코드를 타이핑하는 엄격함이 사라진 것이 아니라, 의도 명세와 결과 검증의 엄격함으로 이동한 것이다.

15. **에이전트가 쉬운 일을 잘한다고 방심하면 안 된다.** (트랙 C의 핵심 교훈) 기초에서 AI가 완벽하더라도, 복잡도가 올라가면 반드시 실패한다.

### 문서 작성 시

16. **트랙 상세 문서는 06_도입부와 동일한 수준의 디테일을 유지하라.** 차시별 타임라인, 구체적 활동, 교사 유의사항(구체적 상황 → 구체적 대응), 예상 문제와 대응을 포함해야 한다.

17. **규칙 파일의 성장 궤적을 추적할 수 있게 하라.** 각 트랙 종료 시 규칙 수의 기대치(~5 → ~15 → ~25 → ~35)와 규칙 예시를 포함해야 한다.

18. **교사 안내는 "이론적 설명"이 아니라 "이럴 때 이렇게 하세요" 형식이어야 한다.** 선생님이 교실에서 즉시 쓸 수 있는 대응 가이드여야 한다.

### 산출물 품질 기준

19. **강의 스크립트를 포함하라.** 선생님이 실제로 할 말을 대사 수준으로 작성한다. 핵심 발문 2~3개와 후속 질문도 포함한다.

20. **학생 예상 반응을 포함하라.** "이때 학생이 이렇게 말할 수 있다", "이 활동에서 흔히 나오는 반응" 등을 구체적으로 기술한다. 선생님이 교실 상황을 미리 그릴 수 있어야 한다.

21. **다양한 예제를 난이도별로 제공하라.** 각 차시의 핵심 활동마다 구체적인 예제를 2~3개 이상 제공한다. 모호한 요청 vs 정밀 명세의 비교, AI 실패 사례와 성공 사례를 나란히 보여준다.

22. **모범 답안을 제공하라.** 명세서, 수용 기준, 규칙 파일 등 학생이 작성해야 하는 모든 산출물에 대해 "잘 쓴 예시"와 "부족한 예시"를 함께 제공한다. 선생님이 채점 기준으로 쓸 수 있어야 한다.

23. **활동지·워크시트·체크리스트를 내장하라.** 별도 파일이 아니라 본문 안에 포함하여, 선생님이 인쇄하거나 화면에 띄울 수 있게 한다.

### 트랙 A(VPython) 설계 변경

24. **트랙 A 차시 1-2의 흐름을 다음과 같이 설계한다:**
    1. 선생님이 귀여운 캐릭터를 제시한다 (별의 커비, 눈사람, 곰돌이 등 — 기본 3D 도형 5~7개로 만들 수 있는 수준)
    2. 학생이 종이에 직접 손으로 따라 그린다 (의도의 원본 확보)
    3. 학생이 직접 VPython으로 코딩해본다 (코딩의 엄격함 체험)
    4. AI에게 같은 것을 만들어달라고 시킨다 (비교 + 규칙의 씨앗)
    5. 직접 코딩하기엔 너무 복잡한 멋진 3D 입체물을 프롬프트로 만들어본다 (AI가 진짜 필요한 순간 + 명세의 위력)

    이 흐름은 Fowler의 "엄격함의 재배치"를 한 차시 안에서 체험시키는 구조다. 코드 타이핑의 엄격함(3단계)이 명세 작성의 엄격함(4~5단계)으로 이동하는 것을 학생이 직접 느낀다.

### 트랙 B(데이터 분석) 설계 방향

25. **트랙 B도 "직접 → AI → 비교 → 복잡한 것은 AI에게" 흐름을 따른다:**
    1. 직감으로 먼저 예측한다 ("우리 학교 급식 만족도는 몇 %일까?" — 내 기준선 확보)
    2. 간단한 데이터를 직접 분석한다 (종이 또는 스프레드시트로 평균, 비율 계산)
    3. AI에게 같은 데이터 분석을 시킨다 (비교: AI가 뭘 잘하고 뭘 놓치는가)
    4. 직접 하기엔 복잡한 대규모 데이터를 AI에게 분석 시킨다 (교차 분석, 시각화 등 — AI가 진짜 필요한 순간)

    핵심은 **"AI가 분석을 잘해도, 해석과 판단은 내 몫"**이라는 체험이다. AI는 상관관계를 인과관계로 표현하고, 표본의 한계를 무시하고, 과잉 일반화를 한다. 이 실수를 잡아내는 것이 규칙 파일의 핵심이 된다. 선생님이 사전 준비한 데이터셋(급식 만족도, 도서관 이용 등) 2~3개를 선택지로 제공하여 데이터 수집의 부담을 줄인다.

### 트랙 C(Pico 2W) 설계 방향

26. **트랙 C는 "AI가 쉬운 일을 잘한다 → 방심 → 복잡해지면 틀린다" 흐름을 따른다:**
    1. 선생님이 간단한 회로(LED 점등)를 직접 보여주고 학생이 따라한다 (기초 체험)
    2. 학생이 직접 MicroPython으로 코딩한다 (LED 켜기, 버튼 제어)
    3. AI에게 같은 과제를 시킨다 → AI가 거의 완벽하게 해결 ("쉬운 건 잘하네")
    4. 점차 복잡한 과제를 AI에게 시킨다 (센서 조합, WiFi 통신 등) → AI가 실수하기 시작
    5. "안 된다!"가 발생하면 원인을 분리한다: HW 문제? SW(AI 코드) 문제? 환경 문제?

    트랙 C의 고유한 교훈은 **"기초에서 AI가 완벽하더라도, 복잡도가 올라가면 반드시 실패한다"**는 것이다. 앞 트랙들에서는 처음부터 AI가 틀렸지만, 트랙 C에서는 처음에 잘하다가 갑자기 틀리기 시작한다. 이 패턴의 차이가 "방심하면 안 된다"는 감각을 만든다. 또한 물리 세계가 대신 검증한다 — LED가 켜지거나 안 켜지므로 해석의 여지가 없다.

    **하드웨어 환경:** Pico 2WH + **Grove 센서 시스템**을 사용한다. Grove 커넥터(4핀 플러그)로 연결하므로 저항 계산, 납땜, 복잡한 배선이 필요 없다. 학생이 하드웨어 연결의 어려움이 아니라 **"AI가 만든 코드가 이 센서에서 동작하는가"**에 집중할 수 있도록 하드웨어 허들을 최소화한다. 브레드보드·저항 배선이 아닌 **"어떤 포트에 어떤 센서를 꽂는가"**가 연결의 핵심이 된다.

### 시각화 전략

27. **트랙별 시각화를 위해 vpython-thinking-30h 프로젝트의 기술 패턴을 재활용한다.** 동일한 5가지 기술 패턴을 3개 트랙에 반복 적용하되, 도메인 특성에 맞게 변환한다:

    | 패턴 | 트랙 A (VPython) | 트랙 B (데이터 분석) | 트랙 C (Pico 2W) |
    |------|:---:|:---:|:---:|
    | **iframe 인터랙티브** | GlowScript 3D 데모 | Plotly.js 차트 | (해당 없음) |
    | **SVG 개념도** | 좌표계 다이어그램 | 차트 구조 / 분석 개념 | **Pico + Grove 포트맵** (어떤 포트에 어떤 센서) |
    | **Mermaid 흐름도** | 애니메이션 루프 | 분석 파이프라인 | **센서→Pico→출력 신호 흐름** |
    | **CSS scene-preview** | 모호 vs 정밀 비교 | 잘못된 차트 vs 올바른 차트 | **Pico + Grove 연결 배치도** |
    | **code-result 블록** | 3D 결과 설명 | 데이터 출력 / 통계치 | **시리얼 출력** (센서 값, 에러 메시지) |

    - 트랙 B: Plotly.js HTML을 `demos/` 폴더에 넣고 iframe으로 임베드 (GlowScript와 동일 패턴)
    - 트랙 C: SVG로 Pico 2WH + Grove 쉴드의 포트 배치를 만들어 "D16 포트에 LED, A0 포트에 온도 센서" 등을 컬러로 표시. Mermaid로 "온도센서→A0→Pico→판단→LED" 같은 신호 흐름을 표현. 브레드보드·저항 배선 없이 Grove 커넥터 기반
    - CSS/JS 인프라를 한 번 세팅하면 전 트랙에서 재활용

28. **`SAMPLE_트랙A_차시1-2_3D조형.md`의 톤/디자인이 전체 교육자료의 기준이다.** 이 샘플에서 확정된 패턴:

## 7. 주의사항

- **총론과 도입부는 이미 높은 완성도**를 가지고 있다. 대폭 수정이 아니라 개선 사항 #1~#5의 정밀 반영이 필요하다.
- **원전 전문은 이 문서에 포함되지 않는다.** 원전의 상세 분석이 필요하면 `07_원전분석_교육과정과의연결.md`를 참조하고, 원전 전문이 필요하면 사용자에게 요청하라.
- **"규칙 파일"은 교실 친화적 용어**다. Hashimoto의 AGENTS.md, Böckeler의 하네스를 모두 이 용어로 통일한 것이다. 문서 작성 시 이 용어를 일관되게 사용하라.
- **API 키는 선생님이 직접 관리**한다. 프록시 서버 방식은 사용하지 않는다.
- **교사 유의사항**은 06_도입부의 스타일을 따른다: 구체적 상황 → 구체적 대응, "이렇게 하세요" 형식.
