## Phase 0: 온보딩 (EntryForm)

### 현황
- Google 로그인 → 소속/주제/방향 입력 → 홈으로 이동

### 문제점
1. **기대 설정 부재**: 사용자가 "이 시스템으로 무엇을 만들 수 있는지" 보기 전에 소속과 주제를 입력해야 함. 아직 시스템의 능력을 모르는 상태에서 주제를 결정하게 됨.
2. **주제·방향 입력의 활용 불투명**: 여기서 입력한 주제가 이후 프로젝트 생성과 어떻게 연결되는지 사용자에게 보이지 않음. 실제로 프로젝트 생성 시 별도로 제목·설명을 다시 입력해야 함.
3. **템플릿 미리보기 부재**: 7개 템플릿이 존재하지만, 온보딩에서는 보여주지 않음. "어떤 종류의 교육자료를 만들 수 있는지"를 먼저 보여주면 사용자의 의사결정이 빨라짐.

### 개선 제안
- 온보딩에서 **샘플 결과물 미리보기** 제공 (예: "이런 교육자료를 만들 수 있습니다" + 7개 템플릿 샘플)
- 주제 입력을 온보딩이 아닌 **프로젝트 생성 시점**으로 이동하거나, 온보딩 입력값을 프로젝트 생성에 자동 연결

## Phase 1: 프로젝트 생성 & 방향 논의

### 1-A. 프로젝트 생성 (/projects)

#### 현황
- 프로젝트 ID, 제목, 작성자, 설명, 대상 독자, 템플릿 선택
- 레퍼런스 업로드 (md, txt, docx, pdf)
- 프롬프트 커스텀 설정
- 빠른 시작 (MD 파일 업로드로 Step 1~3 건너뛰기)

#### 평가
- ✅ **빠른 시작** 기능이 매우 실용적. 기존 교안이 있는 교수자에게 핵심 가치.
- ✅ 레퍼런스 업로드로 기존 자료 활용 가능.
- ⚠️ **템플릿 선택 시점의 정보 부족**: 드롭다운에 이름만 표시. 각 템플릿이 어떤 구조의 결과물을 만드는지 미리보기가 없음.
- ⚠️ **프롬프트 설정 탭**: 일반 사용자가 "목차 생성 프롬프트 추가 지침"이라는 표현을 이해하기 어려움. 이것이 무엇을 제어하는지 예시가 필요.

#### 개선 제안
- 템플릿 선택 시 **샘플 챕터 미리보기** 팝업 (현재 작성한 `template-samples.md` 활용)
- 프롬프트 설정을 "고급 설정"으로 분리하고, 일반 사용자에게는 **가이드라인 입력** (자연어)만 노출

### 1-B. 방향 논의 (/discussion)

#### 현황
- AI와 채팅으로 교육 방향 수립 → 요약 생성 → master-context.md 저장

#### 평가
- ✅ **대화형 설계가 효과적**: 교수자가 자신의 의도를 AI와 대화하며 정리하는 과정은 교육과정 설계의 좋은 출발점.
- ✅ 요약 생성 → 다음 단계 자동 연결.
- ⚠️ **대화 가이드 부족**: 사용자가 "무엇을 이야기해야 하는지" 모를 수 있음. AI가 먼저 핵심 질문을 던져주는 구조가 필요.
- ⚠️ **요약 품질 검증**: 생성된 master-context.md를 사용자가 검토·수정할 수 있는 UI가 없음 (직접 파일 편집은 가능하나 UI 미제공).

#### 개선 제안
- 대화 시작 시 **구조화된 질문 시퀀스** 제공 (대상 학습자는? / 학습 후 기대하는 역량은? / 선호하는 톤은?)
- 요약 생성 후 **편집 가능한 미리보기** UI

## Phase 2: 목차 생성 (/toc)

### 현황
- AI 자동 생성 → JSON 구조 (Part/Chapter/학습목표/개요/예상시간)
- JSON 직접 편집 가능

### 평가
- ✅ **구조화된 TOC 설계**: Part → Chapter → 학습목표 → 개요 → 예상시간 구조가 체계적.
- ✅ 자동 생성 + 수동 편집 모두 지원.
- ⚠️ **JSON 편집 장벽**: 일반 교수자에게 JSON 편집은 심리적·기술적 장벽. 오타 하나로 전체 구조가 깨질 수 있음.
- ⚠️ **estimated_time의 중요도 미인식**: 이 필드가 챕터 분량을 결정하는 핵심 변수인데, 사용자에게 그 영향이 설명되지 않음.

### 개선 제안
- JSON 편집 외에 **비주얼 TOC 편집기** 제공 (카드 드래그, 인라인 편집)
- `estimated_time` 옆에 **예상 분량 표시** (예: "2시간 → 약 7,200~12,000자 생성")

## Phase 3: 피드백 & 확정 (/feedback)

### 현황
- AI와 채팅으로 TOC 수정 → 가이드라인 작성 → 목차 확정

### 평가
- ✅ **AI가 JSON:toc-update 블록을 제안**하는 것은 혁신적. 대화로 수정사항을 반영.
- ⚠️ **가이드라인 입력의 역할 모호**: "콘텐츠 생성 가이드라인"이 무엇을 의미하는지, 어디에 영향을 주는지 사용자에게 불명확. 예시가 제공되지만 UI에서 잘 안 보임.
- ⚠️ **확정의 의미 불명확**: "목차 확정"을 누르면 무엇이 잠기는지, 이후에도 수정 가능한지 안내가 없음.

### 개선 제안
- 가이드라인 입력을 **구조화된 체크리스트 + 자유 입력** 하이브리드로 변경
- 확정 시 **변경 불가 사항 명시** 또는 "언제든 수정 가능"이라는 안내

## Phase 4: 챕터 생성 (/chapters) — 🔴 핵심 이슈 영역

### 4-A. 분량 제어 시스템 — 다층 제약의 충돌

현재 분량은 4단계로 제어:

```
① API maxTokens (기본 12,000)
② _calcMaxTokensForTime() — 30분 이하만 캡 적용
③ 프롬프트 내 "학습 시간 제약" — 글자 수 범위 + "출력 토큰 10,000 이내"
④ TPM 리밋 (분당 토큰 예산)
```

#### 문제점

1. **①과 ③의 충돌**: API maxTokens는 12,000인데, 프롬프트에는 "10,000 이내"라고 지시. 어느 것이 실제 상한인지 모호.
2. **②의 사각지대**: `estimated_time > 30분`이면 캡이 적용되지 않음. 즉 "2시간" 챕터는 12,000 토큰 풀로 생성되지만, 이것이 "2시간 분량으로 충분한지" 또는 "너무 많은지" 검증이 없음.
3. **글자 수 범위의 실효성**: `분×60 ~ 분×100`자 공식이 모든 템플릿에 동일 적용. 그러나 프로그래밍 강의(코드 블록 포함)와 스토리텔링(순수 산문)은 같은 시간이어도 필요 글자수가 다름.
4. **사용자 제어 부재**: 사용자가 "이 챕터는 좀 더 길게/짧게"라고 지정할 방법이 없음. `estimated_time`을 바꾸는 것이 유일한 간접 경로인데, 이 사실이 안내되지 않음.

#### 개선 제안
- ①과 ③의 일관성 확보 (둘 중 하나로 통일하거나 관계를 명확히)
- 템플릿별 글자 수 계수 차별화 (프로그래밍: `분×80~120`, 스토리텔링: `분×70~100` 등)
- 사용자에게 **"분량 프리셋"** 제공 (간결하게 / 표준 / 풍성하게)
- `estimated_time`이 분량에 영향을 준다는 **명시적 안내** UI

### 4-B. 프롬프트 시스템 — 과잉과 경직

#### 문제점

1. **프롬프트 누적 길이**: context_template + toc_prompt_addition + chapter_prompt_addition + chapterGenerator.js 내장 프롬프트 + 시간 제약 + 가이드라인 + 레퍼런스 → **총 입력 토큰이 매우 큼**. Solar Pro 3 같은 소형 모델에서는 프롬프트 자체가 컨텍스트 윈도우를 넘길 수 있음.
2. **이중 프롬프트**: `storytelling.json`의 chapter_prompt_addition과 `chapterGenerator.js`의 isStorytelling 내장 프롬프트가 **유사한 내용을 이중으로** 지시. 어느 것이 우선인지 모델이 혼동할 수 있음.
3. **경직된 퍼센트 제약**: "챕터의 80~90%를 서사로" 같은 지시는 주제에 따라 부적절할 수 있음 (→ 금회 수정 완료).
4. **금지 규칙 과잉**: 스토리텔링 템플릿만 해도 금지 사항이 15개 이상. LLM은 "하지 마"보다 "이렇게 해"에 더 잘 반응하므로, 긍정적 지시 중심으로 재구성하는 것이 효과적.

#### 개선 제안
- **프롬프트 레이어 정리**: JSON 템플릿과 chapterGenerator.js 내장 프롬프트의 역할 분리
  - JSON 템플릿 = "무엇을" (콘텐츠 구조·톤)
  - JS 내장 프롬프트 = "어떻게" (기술적 형식·분량)
- 소형 모델용 **경량 프롬프트 프로파일** 추가 (Solar Pro 3 등에서는 핵심 지시만 포함)
- 금지 규칙을 **긍정형으로 재구성** (❌ "불릿포인트 금지" → ✅ "산문 형태로 개념을 풀어쓰세요")

### 4-C. 멀티 모델 호환성

#### 현황
- 4개 프로바이더 지원: Anthropic, OpenAI, Google, Upstage
- 모든 모델에 동일 프롬프트 적용

#### 문제점

1. **모델별 능력 차이 미반영**: Claude Opus의 출력 품질과 Solar Pro 3의 출력 품질은 상당히 다름. 동일 프롬프트가 소형 모델에서는 지시를 충분히 따르지 못할 수 있음.
2. **컨텍스트 윈도우 차이**: Claude = 200K, GPT-5 = 128K+, Solar Pro 3 = 32K(?). 레퍼런스를 포함한 총 프롬프트가 소형 모델의 한계를 초과할 수 있음.
3. **출력 토큰 상한 차이**: 모델별로 max_output_tokens 상한이 다른데, 현재 코드에서 이를 체크하는 로직이 없음.

#### 개선 제안
- 모델별 **프롬프트 프로파일** (full/compact/minimal)
- `_calcMaxTokensForTime()`에 **모델별 출력 상한 체크** 추가
- 레퍼런스 포함 시 **총 입력 토큰 추정 → 경고** 표시

### 4-D. 생성 품질 검증

#### 문제점
- 생성된 챕터의 **품질을 자동 검증하는 메커니즘이 없음**
- 글자 수가 범위 안에 드는지, 필수 섹션이 포함되었는지, Mermaid 문법이 올바른지 등을 체크하지 않음
- 배치 생성 시 일부 챕터가 잘려도(토큰 초과) 감지하기 어려움

#### 개선 제안
- **생성 후 자동 검증** 단계 추가:
  - 글자 수 범위 체크
  - 필수 섹션 존재 여부 (템플릿별)
  - Mermaid 문법 유효성
  - 불완전 문장으로 끝나는지 (잘림 감지)
- 검증 실패 시 **자동 재생성 또는 사용자 알림**

## Phase 5: 배포 (/deploy)

### 현황
- MkDocs 웹사이트 빌드 → GitHub Pages 배포
- DOCX 내보내기 (pandoc)
- 로컬 미리보기

### 평가
- ✅ **이중 출력이 실용적**: 웹(MkDocs) + 문서(DOCX)로 대부분의 배포 니즈를 충족.
- ✅ 색상 테마 선택 등 커스터마이징 지원.
- ⚠️ **PDF 출력 부재**: 교육 현장에서 PDF는 여전히 핵심 포맷. pandoc → PDF 변환을 추가하면 실용성이 크게 올라감.
- ⚠️ **Mermaid 렌더링**: MkDocs 빌드 시 Mermaid 다이어그램이 정상 렌더링되는지 확인 필요 (플러그인 의존).

### 개선 제안
- PDF 출력 옵션 추가 (pandoc + wkhtmltopdf 또는 weasyprint)
- 배포 전 **최종 미리보기에서 Mermaid 렌더링 검증**

## 템플릿 시스템 — 횡단 평가

### 7개 템플릿의 설계 품질

| 템플릿 | 구조 완성도 | 프롬프트 명확성 | 차별화 | 주요 이슈 |
|--------|-----------|---------------|--------|-----------|
| 스토리텔링 | ★★★★ | ★★★★ | ★★★★★ | 구성 비율 경직 → 수정 완료 |
| 학교 교과서 | ★★★★★ | ★★★★ | ★★★★ | 표준 구조가 잘 정의됨 |
| 프로그래밍 | ★★★★ | ★★★★★ | ★★★★★ | 점진적 빌드업이 핵심 차별점 |
| 자기주도 | ★★★★★ | ★★★★★ | ★★★★ | 트러블슈팅·안전망 설계가 우수 |
| 비즈니스 | ★★★★ | ★★★★ | ★★★★ | 케이스 스터디 품질이 모델 성능에 의존 |
| 교사용 지도서 | ★★★★★ | ★★★★ | ★★★★★ | 가장 구조가 정교함 |
| 워크숍 | ★★★★ | ★★★ | ★★★ | 분 단위 시간 배분이 잘 되어야 하나, AI가 이를 정확히 맞추기 어려움 |

### 공통 이슈

1. **마크다운 테이블 금지 정책의 일관성**: 학교 교과서, 비즈니스에서는 금지하지만, 교사용 지도서에서는 허용. 실제 생성 결과에서 이 규칙이 일관되게 지켜지는지 검증 필요.
2. **Mermaid 의존도**: 모든 템플릿이 Mermaid를 사용하는데, 렌더링 실패 시 사용자에게 코드 블록이 그대로 노출됨.
3. **개념 설명 비중의 유연성**: 스토리텔링 외 다른 템플릿에서도 "주제에 따라 개념 설명 비중을 유연하게 조절" 원칙이 필요.

## 결론

EduFlow는 **교육 콘텐츠 생성 파이프라인의 전 과정(기획→목차→생성→배포)**을 하나의 시스템에 담은 야심찬 프로젝트입니다. 특히 7개 템플릿의 차별화된 프롬프트 설계, AI 대화 기반 방향 수립, 멀티 모델 지원은 높이 평가할 수 있습니다.

가장 시급한 과제는 **Phase 4(챕터 생성)의 분량 제어 일관성과 프롬프트 이중화 해소**입니다. 이 두 가지가 정리되면 생성 품질의 예측 가능성이 크게 올라갈 것입니다.

장기적으로는 **소형 모델 호환성**을 확보하여, Claude Opus 같은 고비용 모델뿐 아니라 Solar Pro 3 등 경량 모델로도 합리적 품질의 콘텐츠를 생성할 수 있도록 하면, 교육 현장의 접근성이 크게 개선될 것입니다.
